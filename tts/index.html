尝试历程与踩坑

❌ 1. 本地运行 IndexTTS 2.0 — 失败

尝试内容：克隆 IndexTTS GitHub 仓库到本地，下载模型权重，尝试本地推理。

坑：

  • 当前环境无 GPU（torch.cuda.is_available() 返回 False），且没有 nvidia-smi 命令。
  • IndexTTS 2.0 依赖 GPU 推理，CPU 模式下无法运行。
  • 模型权重文件（/home/z/my-project/asset/ 下的 safetensors）虽然存在，但没有 GPU 无法加载。

结论：本地推理路线完全不通，必须依赖云端 GPU 服务。

⚠️ 2. HuggingFace ZeroGPU Spaces（IndexTeam/IndexTTS-2-Demo）— 有限可用

尝试内容：通过 gradio_client 库直接调用 HuggingFace 上的官方 IndexTTS-2 Demo Space。

坑：

  • ZeroGPU Spaces 有GPU 配额限制。未认证用户配额极低（约 60 秒 GPU 时间），很快耗尽。
  • 配额耗尽后报错：You have exceeded your ZeroGPU quota (60s requested vs. 0s left)。
  • 需要等待配额重置（通常每小时），或使用 HuggingFace Token 认证获取更多配额。
  • 调用方式：需要使用 handle_file() 传递音频文件，普通文件路径字符串不工作。

关键代码模式：

from gradio_client import Client, handle_file
client = Client("IndexTeam/IndexTTS-2-Demo")
result = client.predict(
    emo_control_method="Same as the voice reference",
    prompt=handle_file("ref_voice.wav"),  # 必须 handle_file！
    text="要合成的文本",
    api_name="/gen_single"
)

结论：可用但配额极低，不适合一次性生成 23 段长文本。

✅ 3. Kozzzq/indextts2api REST API — 最终方案

尝试内容：使用第三方封装的 REST API（https://kozzzq-indextts2api.hf.space），同样是 IndexTTS 2.0 模型。

坑：

  1. API 接口设计：采用异步提交+轮询模式（Submit → Job ID → Poll Status → Download Result），而非同步返回。
  2. Job 过期问题：提交的 Job 如果在约 5 分钟内未被消费，就会被清理（返回 404 Job not found）。因此不能批量提交多个 Job 后统一等，必须逐个提交、逐个等待完成。
  3. 内存问题（OOM Kill）：将参考音频的 base64 编码（约 192KB 字符串）存储在 Python 变量中，加上 requests 库的内存占用，在长时间运行的脚本中会被 Linux OOM Killer 终止。解决方案：每个段落单独读取 base64，及时 gc.collect()。
  4. 后台进程被杀：使用 nohup、setsid 启动的后台 Python 进程，在轮询等待期间经常被系统终止。原因可能是容器的进程管理机制。解决方案：将 submit 和 check 拆分为两个独立的短生命周期脚本，通过 Bash 循环串联。
  5. Tensor 大小不匹配：某些文本偶尔触发模型内部的 tensor size mismatch 错误：The size of tensor a (144) must match the size of tensor b (106) at non-singleton dimension 1。这可能是模型对特定文本长度的 bug，重新提交通常能解决。
  6. 临时文件错误：有时 API 返回 Error opening '/tmp/indextts2-xxx'，这是服务端临时文件被清理导致的，重新提交即可。

最终工作模式：

for i in 0..22:
    1. python3 s.py $i      # 提交 Job，保存 job_id 到文件
    2. sleep 180-300         # 等待 3-5 分钟
    3. python3 c.py $i      # 检查 Job 状态，如果完成则下载音频
    4. 如果失败，重新提交

关键技术细节

参考音频处理

  • 原始音源为用户上传的 M4A 文件（约 150 秒）
  • 需要截取 3 秒片段作为声音克隆参考
  • 使用 ffmpeg 转换为 24kHz 单声道 WAV：ffmpeg -i source.m4a -ss 0 -t 3 -vn -ar 24000 -ac 1 ref_voice.wav

音频生成

  • 23 段文本，每段对应原文的一个自然段落
  • TTS 文本去除换行符，保持语义连贯
  • SRT 字幕保留原始换行格式
  • 每段生成耗时约 1-5 分钟不等
  • 生成的 WAV 格式：PCM 16-bit, 24kHz, Mono

SRT 生成

  • 时间戳基于每个段落音频的实际时长累加
  • 格式：HH:MM:SS,mmm --> HH:MM:SS,mmm
  • 字幕文本保留原始段落格式（含换行）

ASR 验证

  • 使用 faster-whisper (small 模型) 进行语音识别验证
  • 识别语言：中文（概率 1.00）
  • 16 个关键词组中 11 个精确匹配，其余为同音字替换（如"泰娱"→"泰语"，"限泰令"→"现泰令"）
  • 验证结论：音频内容完整、语义正确

Cloudflare Worker 方案建议

哪些步骤可以免费上云？

核心思路：将"提交 TTS Job → 轮询状态 → 下载结果"的流程封装为 Cloudflare Worker，用户只需在网页上传参考音频 + 输入文本，即可自动完成语音合成。

可行的架构

[用户浏览器]
    ↓ (上传音频 + 文本)
[Cloudflare Worker]
    ↓ (转发 API 请求)
[Kozzzq IndexTTS-2 REST API] (免费 HuggingFace Space)
    ↓ (返回音频)
[Cloudflare Worker] → [Cloudflare R2] (存储结果)
    ↓ (返回下载链接)
[用户浏览器]

具体可实现的组件

  1. 前端页面（Cloudflare Pages，免费）：上传参考音频、输入文本、显示进度、下载 WAV 和 SRT
  2. API 代理（Cloudflare Worker，免费额度 10 万次/天）：音频转 base64、提交 Job、轮询状态、下载结果、自动分段、生成 SRT、存储到 R2
  3. 关键注意事项：Worker 30 秒超时需异步架构（Durable Objects / Queues）；建议 D1 数据库记录 Job 状态；ZeroGPU 配额共享需注意并发

为什么可以免费？

核心代码框架（Cloudflare Worker）

// worker.ts
export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const { text, audioBase64 } = await request.json();

    // 1. 提交 TTS Job
    const submitResp = await fetch('https://kozzzq-indextts2api.hf.space/generate', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ text, speaker_wav: audioBase64, language: 'zh' })
    });
    const { job_id } = await submitResp.json();

    // 2. 存储到 D1，让另一个 Worker 轮询
    await env.DB.prepare('INSERT INTO jobs (id, status, text) VALUES (?, ?, ?)')
      .bind(job_id, 'processing', text).run();

    return new Response(JSON.stringify({ jobId: job_id }));
  }
};

局限性

  • HuggingFace Space 并非 100% 可用，偶尔会因配额或维护不可用
  • Cloudflare Worker 的 30 秒超时限制了同步等待，需要异步架构
  • 长文本需要分段处理，每段约需 1-5 分钟生成时间
  • 声音克隆质量受参考音频质量影响很大（3-10 秒清晰语音最佳）
